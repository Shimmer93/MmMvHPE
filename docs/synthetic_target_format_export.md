# Synthetic Target Format Export

This document describes the exporter that turns an existing SAM-3D-Body synthetic sample into training-facing GT bundles for MMHPE.

## Scope

The exporter does not create HuMMan-like or Panoptic-like dataset trees.

It writes only the minimal target-format artifacts needed for downstream training under each existing synthetic sample directory:

- `exports/humman/`
- `exports/panoptic/`
- `exports/export_manifest.json`

This avoids duplicating source images, masks, and point clouds that are already stored by the base synthetic generation pipeline.

## Current Targets

### HuMMan Export

The HuMMan-oriented export writes:

- `gt_keypoints.npy`
  - SMPL24 keypoints in new-world coordinates
- `gt_keypoints_world.npy`
  - SMPL24 keypoints in world or camera coordinates before new-world normalization
- `gt_smpl_params.npy`
  - 82D SMPL payload: pose(72) + betas(10)
- `rgb_camera.json`
- `lidar_camera.json`
- `gt_keypoints_lidar.npy`
- `gt_keypoints_pc_centered_input_lidar.npy`
- `gt_keypoints_pc_center_lidar.npy`
- `input_lidar.npy`
  - symlink to the original synthetic LiDAR point cloud

Optional derived outputs:

- `gt_camera_rgb.npy`
- `gt_camera_lidar.npy`
- `gt_keypoints_2d_rgb.npy`
  - normalized to `[-1, 1]` and clipped to that range to match the HuMMan and Panoptic dataset loaders

### Panoptic Export

The Panoptic-oriented export writes:

- `gt_keypoints.npy`
  - Panoptic joints19 in new-world coordinates
- `gt_keypoints_world.npy`
  - Panoptic joints19 in world or camera coordinates
- `rgb_camera.json`

Optional derived outputs:

- `gt_camera_rgb.npy`
- `gt_keypoints_2d_rgb.npy`
  - normalized to `[-1, 1]` and clipped to that range

Panoptic export is intentionally keypoint-only and does not save `gt_smpl_params`.

## Skeleton Conventions

### HuMMan

HuMMan export uses SMPL24 joints:

- pelvis root at index `0`
- `gt_keypoints.npy` is new-world and root-rotation-removed
- `gt_keypoints_world.npy` stays in the pre-normalized world or camera frame

### Panoptic

Panoptic export uses the repo's Panoptic joints19 contract:

- neck at index `0`
- body center at index `2`
- left hip at index `6`
- right hip at index `12`

The exporter derives Panoptic joints19 directly from rerun MHR70 keypoints:

- `neck` from MHR70 index `69`
- `nose` from MHR70 index `0`
- `body_center` from the midpoint of left and right hips
- shoulder, elbow, wrist, hip, knee, ankle, eye, and ear from their direct MHR70 counterparts

## Camera Contract

The exporter writes MMHPE-style camera dicts:

- `intrinsic`: `(3, 3)`
- `extrinsic`: `(3, 4)` world-to-camera

It can also save `gt_camera_<modality>` pose encodings using the existing `absT_quaR_FoV` encoding path.

For `gt_camera_lidar.npy`, the exporter mirrors the current repo transform semantics:

- the pose encoding FoV is derived from a pseudo frame size inferred from LiDAR tensor shape
- for a saved single-frame point cloud `input_lidar.npy` with shape `(N, 3)`, the exporter uses `(H, W) = (1, N)`
- this matches `CameraParamToPoseEncoding` when LiDAR is provided as `(T, N, 3)` with `T = 1`

## MHR To SMPL Conversion

The exporter requires an MHR checkout because it uses the upstream MHR conversion mapping asset:

- `tools/mhr_smpl_conversion/assets/mhr2smpl_mapping.npz`

Reference:

- <https://github.com/facebookresearch/MHR/blob/main/tools/mhr_smpl_conversion/README.md>

Current implementation details:

- reruns SAM-3D-Body for each exported sample to recover `pred_cam_t`, `focal_length`, and full MHR outputs
- prefers the official MHR conversion backend with the upstream `Conversion(..., method="pytorch")` path
- keeps a local barycentric-mapping fallback if the official backend is unavailable
- records the selected backend and fitting errors in the export manifest

This keeps the exporter usable in MMHPE today while still preferring the upstream conversion path when the runtime is available.

## Requirements

You need:

- existing synthetic samples generated by `scripts/run_sam3d_synthetic_v0a*.py`
- SAM-3D-Body checkpoints
- a local MHR checkout with `tools/mhr_smpl_conversion/assets/mhr2smpl_mapping.npz`
- the MHR release asset bundle unpacked under `<mhr_root>/assets/`
  - `compact_v6_1.model`
  - `lod1.fbx`
  - `corrective_blendshapes_lod1.npz`
  - `corrective_activation.npz`
- Meta's `pymomentum` runtime
  - `pymomentum-gpu` on CUDA machines, or `pymomentum-cpu` otherwise
- a usable SMPL model file

Example local assets used during validation:

- MHR repo: `/tmp/MHR`
- SMPL model: `weights/smpl/SMPL_NEUTRAL.pkl`

## Run One Sample

```bash
uv run --no-sync python scripts/export_synthetic_gt_formats.py \
  --synthetic-root /opt/data/coco/synthetic_data/v0a_train2017 \
  --sample-dir /opt/data/coco/synthetic_data/v0a_train2017/ann_000000183014_img_000000045687 \
  --mhr-repo-root /tmp/MHR \
  --smpl-model-path weights/smpl/SMPL_NEUTRAL.pkl
```

## Run A Small Batch

```bash
uv run --no-sync python scripts/export_synthetic_gt_formats.py \
  --synthetic-root /opt/data/coco/synthetic_data/v0a_train2017 \
  --mhr-repo-root /tmp/MHR \
  --smpl-model-path weights/smpl/SMPL_NEUTRAL.pkl \
  --start-index 0 \
  --max-samples 8
```

Resume a larger export:

```bash
uv run --no-sync python scripts/export_synthetic_gt_formats.py \
  --synthetic-root /opt/data/coco/synthetic_data/v0a_train2017 \
  --mhr-repo-root /tmp/MHR \
  --smpl-model-path weights/smpl/SMPL_NEUTRAL.pkl \
  --start-index 0 \
  --max-samples 1000 \
  --resume
```

## Output Layout

Example:

```text
ann_<annotation_id>_img_<image_id>/
  manifest.json
  arrays/
  source_mask.png
  exports/
    export_manifest.json
    humman/
      input_lidar.npy -> ../../arrays/synthetic_lidar_points_sensor.npy
      gt_keypoints.npy
      gt_keypoints_world.npy
      gt_smpl_params.npy
      rgb_camera.json
      lidar_camera.json
      ...
    panoptic/
      gt_keypoints.npy
      gt_keypoints_world.npy
      rgb_camera.json
      ...
```

## Visual QC

Render a small QC gallery from already exported samples:

```bash
uv run --no-sync python scripts/visualize_synthetic_gt_exports.py \
  --synthetic-root /opt/data/coco/synthetic_data/v0a_train2017 \
  --output-dir logs/synthetic_data/export_qc_v1 \
  --start-index 0 \
  --max-samples 4
```

This writes:

- one figure per selected sample
- `index.png` as a contact sheet for quick review

Each sample figure includes:

- source RGB
- saved mask with bbox
- HuMMan 2D RGB projection
- Panoptic 2D RGB projection
- MHR70 world skeleton
- HuMMan SMPL24 world skeleton
- Panoptic19 world skeleton
- world-space synthetic LiDAR point cloud overlaid with MHR70 / SMPL24 / Panoptic19
- conversion and sensor metadata

## Validation Notes

The exporter was validated on a generated COCO-train synthetic sample by:

- running one-sample export end to end
- checking `gt_camera_*` outputs against the current pose-encoding path
- checking `gt_keypoints_pc_centered_input_lidar` against `PCCenterWithKeypoints`
